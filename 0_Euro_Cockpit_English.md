Chapter 1: Data Privacy & Local AI

Before we get started, we need to talk about the foundation: Your data. As the aspiring "lead developer" of your company, you bear the responsibility for what moves to the cloud and what remains securely in your own vault.

The GDPR (General Data Protection Regulation) is often perceived as a "bureaucracy monster". At its core, however, it is just a checklist for professional work. It states: You may only process personal data if you have a legal basis.

When are you allowed to upload personal data?

Consent: If your customers or employees have explicitly agreed (e.g., through an addendum in your T&Cs or contracts).

Legitimate interest: For purely organizational processes without sensitive details (like finances), this is often justifiable – although, technically speaking, the data still leaves your direct control upon upload.

This presentation does not replace legal advice. In case of doubt, consult your data protection officer!

To ensure your cockpit doesn't become a data leak, you should use the following options:

1.1 Anonymization

Use the oldest trick in IT: Search & Replace (Ctrl + H). Swap "Müller GmbH" for "Customer_A" or "Ms. Schmidt" for "Contact_Person_1". The AI doesn't need real names to understand logic – it only needs the structure. What is not uploaded cannot be leaked.

1.2 Digital Muzzle 

Almost all major AI providers use your chats to train their models by default. You don't want that for your business. Here is how you prevent it:

The ChatGPT Check:

Go to Settings -> Data controls and disable "Chat history & training".

Pro tip: Use the "Temporary Chat". Here, inputs are not used for either training or personalization.

The Google Check (Gemini & AI Studio):

Gemini (Web): Go to gemini.google.com, open "Activity", and turn off "Save activity".

Google AI Studio (aistudio.google.com): Check the data processing options in the settings. Pay attention to whether your data may be used for "product improvement" and actively opt out of it.

Workspace (Gmail/Drive): In the Gmail settings under "Smart features and personalization", disable data usage for Google services.

Beware the GDPR Trap: Just because "training" is turned off doesn't mean the system is automatically 100% GDPR-compliant. Data often flows through US servers. For highly sensitive customer data, this is only the second-best solution.

Ask yourself the following question before every upload: Would it be bad if this information were in the newspaper tomorrow?

If YES: Use a local AI or anonymize consistently before copying.

If NO: Use the cloud, but make sure training is disabled.

1.3.2 Which Model is Right for Me?

The app alone is just the chassis – you still need the "engine", the so-called model. The numbers behind the names (e.g., 3B, 7B, 14B) describe the size of this engine (parameters in billions). Before we get started, let's briefly find the right model for your use case:

3B to 4B: Nimble sprinters. They run smoothly on almost any modern laptop and are perfectly sufficient for most cockpit tasks (our all-rounder tip: qwen3.5:4b).

7B to 9B: The "gold standard". Noticeably smarter with complex contexts, but require more computing power.

14B+: The heavyweights for deep data analysis.

You can find an overview of all available models and their run commands directly in the Ollama Model Library (ollama.com/library). Most models come in various sizes. Modern models like Qwen 3.5 (ollama.com/library/qwen3.5) or Gemma 4 (ollama.com/library/gemma4) are very efficient, intelligent, and multimodal – meaning they can also process images.

The Hardware Check: You don't need an office data center for local AI. As little as 16 GB of RAM and a graphics card with 4 GB of VRAM (e.g., GTX 1650) are enough to run the small models (3B to 4B) stably. If you want to work smoothly with the larger models, 8 GB of VRAM (e.g., RTX 3060) or a current Apple Mac with an M-chip and unified memory are exactly what you need.

1.3.4 The Hybrid Strategy

Finally, we put the puzzle pieces together. Why the effort with Ollama and the terminal? Because we run a hybrid strategy in a business context. Security meets speed – utilize the strengths of both worlds:

Sanitize locally: You feed your actual raw data into Ollama locally on your computer (e.g., via the desktop app) and have it anonymized in compliance with data privacy regulations.

Think big: You then copy these clean, "harmless" facts into the powerful cloud models (like Gemini or ChatGPT) without any data privacy concerns, to let them generate the complex content for your cockpit.

This way, you maintain full control over sensitive data at all times, without having to forgo the computing power of the tech giants!

1.4 The Most Common Mistakes and How to Avoid Them

Overburdening hardware

The problem: A massive 14B model brings a standard laptop to its knees. The fans howl, the system freezes, and answers arrive at a snail's pace.

The solution: Always start with lean models in the 3B to 4B class (e.g., qwen3.5:4b). Only switch to larger models when the performance is no longer sufficient for the task. You can find models in the Ollama Library (ollama.com/library).

Drag-and-Drop in the terminal

The problem: Dragging Word or PDF files directly into the command line window doesn't work and generates errors.

The solution: Use copy-and-paste for plain text in the terminal, though depending on the model, you can also drag images in. For processing entire documents, you should use a graphical interface like Open WebUI (openwebui.com).

Data without clear guidance 

The problem: You throw raw data at the AI without telling it what to do. The AI guesses, loses the thread, and begins to "hallucinate" (invent things).

The solution: Always state the role and instruction first (e.g., "You are a data privacy assistant. Anonymize the following text:"). Visually separate the command and the data, ideally with dashes (---).

Blowing up the AI's memory

The problem: You copy 50 pages of text into the chat at once. The AI ignores the beginning or breaks off in the middle because its "short-term memory" (Context Window) is full.

The solution: Feed the model in digestible chunks and break large datasets down into smaller sections.

Ignoring the language barrier

The problem: You write in German, but the AI answers clumsily or switches to English. Many models are primarily trained on English.

The solution: Specifically use multilingual models that, even in compact variants, have excellent command of German.

2.1 Mapping Processes

From the initial contact to the receipt of payment, every company is carried by its processes. In reality, however, it often looks like this: These structures primarily exist as experiential knowledge in someone's head or gather dust in outdated files.

By the way, did you notice the third arm in the picture? Always remember: AI can make mistakes!

The problem: Shoving pixels and "Doc Rot". Anyone who has ever tried to build a multi-stage flowchart in conventional office programs knows the pain: You move one element and the entire layout breaks. With every minor adjustment in the operational workflow, the time-consuming process of dragging and dropping boxes starts all over again. This inevitably leads to the aforementioned "Doc Rot" - documentation becomes outdated faster than you can draw it, or have the desire to do so.

The solution: "Diagram-as-Code". We no longer touch PowerPoint for processes. Instead, you describe the workflow in simple text. A software reads this code and generates the diagram automatically. This saves you not only the annoying alignment of boxes, but above all the classic, tedious modeling work. If a step changes, you only adjust the corresponding word in the text (or code) – the layout updates immediately and aligns itself automatically. Your processes can be versioned, are always cleanly structured, and form the foundation for true automation.

Your Tools I have selected two free open-source tools that implement this text-to-diagram approach perfectly:

Mermaid (mermaid.js.org): Transforms text (or simple code) into diagrams in the browser. Ideal for fast flowcharts and logical workflows. Additionally, the lightweight code can later be embedded seamlessly and without loading delays directly into your HTML cockpit.

BPMN.io: This tool relies on the global industry standard BPMN (Business Process Model and Notation, version 2.0). Here you work with swimlanes (responsibilities) and gateways (branches) – exactly what you need when processes must be documented in a way that is audit-proof, scalable, and binding.

2.2.2 The C4 Model 

The rescue from this chaos is an IT development plan. The C4 model (c4model.com) is suitable here. C4 stands for: Context, Container, Components, Code. A single glance at such a context diagram in your dashboard immediately shows you all systems and their dependencies. Let me explain it to you:

Just imagine a Google Maps map:

1. Context (The globe): Your farm in the middle and all the external systems around it that it communicates with.

2. Container (The city map): You zoom into the system. Now you see the rough building blocks: the web app, the database, and the mobile app. Here you can see which technologies (e.g., Java or SQL) are used.

3. Components (The building plan): You zoom into a container (e.g., the web app). Now you see the individual functional blocks inside, such as the "login service" or the "email dispatch".

4. Code (The floor plan): This is the magnifying glass. Here you see the actual classes or functions. In practice, this level is often omitted because it changes too quickly.

Practice shows: For most small and medium-sized enterprises, this first level – the system context – is completely sufficient as a strategic basis. The deeper levels (Container, Components, Code) usually only become relevant when you have your own software developed. 

What we build here in a few minutes via prompt is a building block of Enterprise Architecture Management (EAM) – and without the otherwise usual, tedious manual system and modeling work. 2.5 The Most Common Mistakes and How to Avoid Them

Before we jump into the next chapter, here is some quick troubleshooting. When you handle generated code, something will inevitably get stuck at some point. That is completely normal and part of the process. With these four tricks, you can save the situation in seconds:

Error messages instead of diagrams: You pasted the code, and instead of a chic diagram, the editor just spits out error messages? Don't panic. You don't have to learn to debug this code yourself. 

Solution: Simply copy the entire error message, throw it into the chat with your AI and say: "There was an error here, please correct it." The machine will apologize and deliver the repaired code to you.

The special character diva: Mermaid in particular can sometimes react extremely cranky to unexpected brackets or exotic special characters in the text. If the layout suddenly breaks, this is often the cause. 

Solution: Simply enclose the affected text in the code in double quotes. The AI theoretically knows this itself, but sometimes forgets – here you are the supervisor.

The secret update: You open your cockpit and suddenly the diagrams are gone? Tools like Mermaid are evolving rapidly. If you always load the latest version via the CDN (Content Delivery Network) in your HTML code, an update from the publisher can break your code.

Solution: Always ask the AI in your prompt to include a fixed version (e.g., 11.x). Just research this online beforehand.

The monster diagram: Don't even try to squeeze your company's entire workflows of the last ten years into a single chart. The result resembles an exploded circuit diagram and is not helpful in everyday life. 

Solution: Start small! A process with 5 to 7 individual steps is perfect. If it gets more complex, cut the process and outsource the rest to a second diagram.

The overzealous assistant (AI hallucinations): When faced with confusing data, AIs often creatively fill in gaps instead of admitting ignorance. Such invented values falsify your dashboard and lead to wrong decisions. 

The solution: Build a "seatbelt" into your prompt: 

IMPORTANT: If you don't understand data or find gaps, do not guess. Do not fill in anything on your own, but ask me specifically.

This prevents blind guessing and secures your data quality.

Data loss in the browser: Web editors like the Mermaid Live Editor, the PlantUML Web Server, or BPMN.io do not save reliably. If you accidentally close the tab or reload the page, your masterpiece has often vanished into digital nirvana. 

Solution: Make it a habit to always save the generated text code in parallel in a simple local text file, or export the finished diagram immediately as an image.

If several things don't fit, change them individually as well. Test every small adjustment immediately in JSFiddle.

The wrong file extensions. Your browser only shows confusing text instead of a cockpit? Often file extensions are hidden in the system and you unknowingly save a pure text file.

Solution: Exactly as discussed in Chapter 3.4: Make the file extensions visible in your system so that your secret .txt finally becomes a real .html.

Fear of the code. You don't dare to change a single word in the HTML, for fear of destroying the entire system.

Solution: Use your sandbox at JSFiddle. You can break absolutely nothing there. Pressing Ctrl + Z (Undo) repairs any mistake immediately. Just go for it!

The column name trap during dropzone upload. Your exported Excel file suddenly calls the column "Amount (EUR)", but your code stubbornly looks for "Amount". The script finds nothing, and the diagrams remain empty after dragging the file in.

Solution: Always keep the exact same column names in your exports from the bank or CRM. Alternatively: Ask the AI in your prompt to recognize the column names in the code flexibly ("fuzzy"), so that variations are ignored. 

The comma-semicolon catastrophe. German Excel versions often export CSV files with a semicolon as a separator, but international tools (like PapaParse) expect a comma by default. After the drop, nothing happens at all.

Solution: It is best to save your tables directly as a regular Excel file (.xlsx); this format problem does not exist there. If you must use CSV, add this to your prompt: 

"Configure PapaParse to automatically detect delimiters."

The invisible date trap. Excel often saves dates internally as sequential numbers (e.g., 45627 instead of 15.11.2026). After the drop, your cockpit then displays completely confusing numbers on the chart axes.

Solution: If you want to evaluate dates in the cockpit, build this sentence into your master prompt as a safeguard: "Automatically convert Excel date formats into a readable format (ISO) when reading them out."

The AI breaks off in the middle: The AI generates your code, but suddenly stops in the middle of a line or bracket.

The solution: This is not an error; the AI has simply reached its maximum response length (the so-called token limit). Do not click "Regenerate"! Write in the chat: 

Continue writing the code exactly from the point where you left off.

Then seamlessly copy both code parts together into your text file.

Local (Ollama): Here you don't need a classic key, as the AI safely resides on your own computer anyway. But for your local browser cockpit to be allowed to speak with Ollama at all (without being blocked by the browser's internal security rules), we use the universal start command with the asterisk (*):

OLLAMA_ORIGINS="*" ollama serve

The asterisk (*) in this command is extremely convenient, but it also means that, theoretically, any website with malicious intent that you currently have open in your browser could access your local AI. When you finish work, you take the key out again – you simply press Ctrl + C in your black window and close it. As soon as Ollama is terminated, your computer is 100% "locked" again.

The Offline Trick: If that's too sketchy for you, there is a safe way: Briefly disconnect your computer from the Wi-Fi or unplug the network cable for the time you are working on your cockpit. Where there is no internet, there is no danger from the outside.

6.1 Architecture Update

The current cockpit.html bundles everything into a single file. This is technically necessary for getting started: If you open the cockpit locally by double-clicking it, the browser would block multiple separate HTML files for security reasons. But this monolithic block becomes confusing over time. If you just want to make a small change, you have to scroll through thousands of lines of code – that costs nerves and creates sources of error.

The solution is the "divide and conquer" principle. There are two approaches here, depending on your deployment location.

6.1.1 Modularizing Code (for Local Use)

Since we cannot cut it into real files for local use, we instruct the AI to sort the code within the single file into isolated modules – like drawers in a cabinet. This is strictly separated into three levels:

Layout (HTML): At the top lies the basic framework – header, navigation, content area, and footer. This "picture frame" remains unchanged.

Design (CSS): All formatting is bundled centrally in the <style> area. If you want to change the color scheme, you don't search through the whole document, but only adapt this one block.

Function (JavaScript): The logic – navigation, API calls, interactions – resides in the <script> area. This is clearly separated from the content.

Data Blocks: At the end of the file, all content goes into clearly commented sections. Each area is marked with instructions like <!-- INSERT YOUR DATA HERE -->.

Using the cockpit.html as an example, the blocks would look as follows:

HTML: Defines the layout and the basic structure of the page.

CSS: Controls the design, colors, and visual presentation.

JavaScript: Contains the interactive functions and application logic.

Data Blocks: Contains the actual content, clearly commented.

The advantage: You still only have a single file. You can put it on a USB stick, send it by email, or open it locally – it works immediately on any computer with a double-click. No browser blocks, no server configuration.

The advantage: You still only have a single file. You can put it on a USB stick, send it by email, or open it locally – it works immediately on any computer with a double-click. No browser blocks, no server configuration.

The disadvantage: As the scope grows, the file becomes confusing. Teamwork is difficult since everyone is working on the same document. External automation tools cannot access individual data.

For whom? Beginners, solo users, and anyone who wants to run the cockpit locally without a server.

6.1.2 Dismantling the Cockpit (for Server Hosting)

As soon as your cockpit is on a server – be it on GitHub Pages, a classic web hosting provider, or on your own Raspberry Pi – the browser's technical restrictions no longer apply. Via HTTP or HTTPS, you can then easily load separate CSS, JavaScript, and HTML files.

The advantages at a glance:

Maintainability: Changes to the design only affect the CSS file, logic changes only the JavaScript file. You no longer have to search through a huge HTML file.

Teamwork: Colleagues can work on individual pages without destroying each other's code. Everyone takes care of their module.

Load time: The browser only loads the files needed for the current view. This makes your cockpit noticeably faster.

Expandability: You add new functions as a separate file – without touching the basic framework. Do you want a ticket page? Simply create tickets.html.

Automation: External tools like n8n or OpenClaw can update CSV files in the same folder without touching the HTML structure. This is the key to true automation.

6.2 Providing the Cockpit Online

Your cockpit works perfectly locally. But if you work in a team, you all want to access the same, up-to-date data. The central command must therefore go online.

Before you begin, you must know an important ground rule of the internet: If you want to call up your cockpit as a website, you absolutely must rename your main file to index.html. Web servers are programmed to load exactly this file as the start page. If your file is named something else (like cockpit.html), the server will not recognize it and your screen will remain blank.

Now we face a problem well-known in the industry: Simply uploading a website is often free today – but as soon as you want to put a secure "door lock" (password protection) in front of it, almost all providers suddenly demand money.

An extremely important warning upfront: If you ask an AI how to password-protect an HTML file for free, it will often advise you to use a small JavaScript code that displays a password window upon opening. Never do this! This is purely "fake protection". Anyone who right-clicks in the browser and selects "View Page Source" immediately sees all your financial and process data, as well as the password, in plain text.

To make your cockpit truly secure and provide it to your team for free, there are these solutions, depending on your technical comfort level:

6.2.1 The Digital Vault (GitHub for 0 Euros)

GitHub is the world's largest platform for software projects. For no-coders, this is our first choice when it comes to pure data exchange.
With free GitHub accounts, you can store files extremely securely and privately in the cloud.

Step 1 (Create Vault): Create a new project (a so-called "repository") and be sure to set it to "Private". This protects everything from unauthorized access.

Step 2 (Invite Team): Under the settings ("Settings > Collaborators"), invite your colleagues via email. Only they will receive the key to the vault.

Step 3 (The Workflow): Your colleagues log in to GitHub, download the current cockpit file, and open it locally on their computer with a double-click.

Important note: Never use the "GitHub Pages" function with GitHub! Even for a private project, this function publishes your site visibly to everyone on the open internet. Access control for websites ("Access Control for GitHub Pages") is only available in the GitHub Enterprise Cloud tier (approx. €20 per month).

Conclusion: 0 Euros, 100% GDPR-compliant. GitHub serves here as a highly secure, digital corporate USB stick.

6.2.2 Cloudflare for 0 Euros – Live on the Web

If constant downloading is annoying and your team should be able to open a real live link (e.g., your-cockpit.pages.dev) in the browser, we use a trick: We combine two free services from the provider Cloudflare (cloudflare.com):

Cloudflare Pages: Here you simply upload your files via drag-and-drop. Cloudflare turns them into a lightning-fast website for free.

Cloudflare Zero Trust: This is our bouncer. This enterprise security feature is completely free for up to 50 users. You specify in the menu that only certain email addresses (e.g., @yourcompany.com) are allowed to access the site.

The Implementation

Since Cloudflare was actually built for IT professionals, the menus can seem a bit intimidating at first glance. But don't worry, just follow this recipe:

The Hosting: Create a free account on cloudflare.com. In the menu on the left, click on "Workers & Pages" and select "Upload assets". Drag your folder with the index.html into the field. Cloudflare will now generate your live link (e.g., https://cockpit-abc.pages.dev).

Activate Zero Trust: Click on "Zero Trust" in the main left menu. Important note: Cloudflare will ask you for a payment method here (credit card/PayPal) and you must choose the "Free Plan". Don't panic! This is only to ward off spam bots. 0 euros will be deducted.

Set up the bouncer: In the Zero Trust menu, click on "Access" and then on "Applications". Click on "Add an Application" and select "Self-Hosted". Enter the link you created earlier (your-name.pages.dev) here.

Write the guest list: In the next window, scroll down to the "Policies" (the rules). Here you determine who is allowed in. Select "Allow" as the action and insert the email addresses of your employees under "Include" (or simply your entire company domain, e.g., @yourcompany.com).

Testing: Save everything. Now open an "Incognito window" in your browser and open your link. If you did everything right, instead of your cockpit you will now see the prompt: "Please enter email address to receive an access code."

Tip for No-Coders: If menus at Cloudflare have changed in the meantime, just take a screenshot of the Cloudflare page, upload it to your AI, say you're stuck, and you will be helped!

The Workflow: If an employee accesses your link, the page is blocked. They have to type in their email address and immediately receive a PIN code in their inbox. If they enter this, the cockpit opens. No password to remember or that can be hacked – absolute enterprise-level security for 0 euros!

6.2.3 Web Hosting

Perhaps you already rent a server for your company website anyway (with providers like IONOS, Strato, or Hostinger). In this case, there are no new costs.

Uploading: You simply push your files via a free file transfer program (like FileZilla – your digital mover) onto your server into a subfolder.

The Door Lock: Since the internet is open to everyone, it is mandatory to protect this folder. With web hosts, this is done reliably using a tiny text file called .htaccess.

Tip for No-Coders: Simply ask your AI: "I have a web space. Write me the .htaccess code to protect my cockpit folder with the password 'Secret123', and explain to me step by step how and where I have to create this file."

6.2.4 Self-Hosting (The Nerd Route)

6.2.4 Self-Hosting (The Nerd Route)

You don't want your data in any cloud, not even with Cloudflare or your web host? Then you have to become the server provider yourself. An honest warning upfront: At this point, we are leaving the pure no-code area and venturing into the world of networks. Let the AI explain every step of this to you!

Variant A: Hosting from your own laptop (The company Wi-Fi)
Can you share your cockpit directly from your work laptop? Yes. You use the Python trick from Chapter 5 (the command python -m http.server 8000). As long as this black command line window is open, your laptop broadcasts the cockpit into your local network. When the colleague at the desk opposite you types your IP address into their browser, they see the cockpit.

Attention, dangerous data leaks lurk here:

Wrong folder: Never start this command in your main folder (like "My Documents"). Otherwise, anyone in the Wi-Fi can download your private invoices or vacation photos via their browser! Always create an isolated folder exclusively for the cockpit.

Wrong Wi-Fi: Only use this trick in the secure office Wi-Fi. If you do this on a high-speed train, in a café, or on an open guest network, you are presenting your company figures on a silver platter to every stranger on the same network.
(Additional warning: Never try to reprogram your home router so that colleagues can access your PC via the open internet. You are tearing a hole in your firewall that hackers will be thrilled about!)

Variant B: The Raspberry Pi (The Supreme Discipline)
If you want full control but don't want to leave your work computer running 24 hours a day, buy a Raspberry Pi (approx. 80 Euros). It is a full-fledged mini-computer the size of a credit card.

You connect it to your router, install a web server, and put your cockpit on it. With tools like "Tailscale" (an extremely beginner-friendly, free network tool), you then stretch a kind of invisible cable between your employee's laptop in the home office and your Raspberry Pi in the office.

Conclusion: One-time hardware costs, 0 Euros per month. It requires some familiarization with the AI, but it is the ultimate project for newly minted system architects who want 100% data control!

OpenClaw as your digital webmaster: If you have placed your cockpit on GitHub as described earlier, OpenClaw unleashes its true potential. You no longer have to export and upload data manually.

This fully automated Git integration brings true enterprise architecture level to your projects – almost as seamlessly and invisibly. In short: The AI does all the prep work and follow-up.

0–9

0-Euro Cockpit: A self-built, free digital command center for one's own company, which manages without expensive licenses and merely requires a web browser and text editor.

CORS (Cross-Origin Resource Sharing): A security mechanism in the browser. It can trigger error messages if your local HTML file tries to grab data directly from a foreign API (like a cloud AI provider).

CRM (Customer Relationship Management): Software for managing customer relationships – e.g., for sales, customer service, and contact maintenance. Centrally stores who your customers are, what they have bought, and what communication has taken place.

CSV (Comma-Separated Values): A simple, text-based file format for storing tabular data. Ideal for transferring export data from banks or CRM systems to an AI.

D

Diagram-as-Code (DaC): A method where flowcharts or IT architectures are not painstakingly drawn by mouse, but described in simple lines of text. The software generates and updates the final image from this fully automatically.

Doc Rot (Documentation Decay): The phenomenon where manually created graphics and processes become outdated extremely quickly because the effort to change them in programs like PowerPoint is too high in everyday life.

Diagram-as-Code: Which tool for what?

Instead of drawing, you describe the structure as text. The AI generates the code, the browser paints the picture.

Tip for design adjustments: Import generated code into diagrams.net (app.diagrams.net) and refine it via drag-and-drop.

The Cockpit Foundation 

Your cockpit is a single, local HTML file. You simply edit it in a text editor.

Testing without saving: Use JSFiddle (jsfiddle.net) to test code changes in real-time and with absolutely zero risk (Ctrl+Z saves everything).

Local Updates (Dropzone): Use libraries like SheetJS and PapaParse via CDN to read CSV and Excel files locally in the browser without data leaving your system.

API Interfaces (The "Waiter"): Use the JavaScript fetch() function to pull data from an AI into your cockpit.

JSON constraint: Always use the following addition in the prompt: "Answer EXCLUSIVELY in JSON format, no introduction."

Hosting & Sharing : 

Variant A (Secure Vault): Upload your cockpit to a private GitHub repository. Share it only via the "Collaborators" function with your team. Colleagues download the file and open it locally.

Variant B (Live link with Cloudflare): For a real web link, use Cloudflare Pages in combination with Cloudflare Zero Trust.

Attention GitHub Pages: Only activate GitHub Pages if your cockpit contains absolutely no sensitive data (e.g., only public marketing). With free GitHub accounts, Pages websites are often publicly viewable, even if the repository is private!

Quick Troubleshooting (The 4 Rescuers)

Error messages in the diagram: Don't debug it yourself! Copy the error message, paste it into the AI and say: "There was an error here, correct it."

Browser shows code instead of dashboard: Your file is secretly named cockpit.html.txt. Show file extensions in the operating system and delete the .txt!

CORS Blockage (Console shows error): Your browser is blocking the cloud API for security reasons. Either use a local AI (Ollama) or OpenRouter (openrouter.ai), which offer CORS-friendly endpoints.

Layout breaks in Mermaid: The AI choked on special characters. Simply put text passages in the code in double quotes "...".
